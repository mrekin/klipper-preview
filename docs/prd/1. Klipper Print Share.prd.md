# 1. Klipper Print Share

Status: PRD_READY

## Контекст / идея

Сервис Klipper Print Share предназначен для операторов 3D-принтеров, работающих с Klipper, которым необходимо демонстрировать заказчикам прогресс печати без предоставления доступа к управлению принтером.

Система предоставляет возможность генерировать временные ссылки на просмотр статуса 3D-печати. По ссылке заказчик видит:
- Прогресс печати в процентах и слоях
- Текущее время печати и расчётное время завершения (ETA)
- Температуры сопла и стола
- Интерактивную визуализацию G-code (траектория движения головки, текущий слой)

Ссылки имеют настраиваемое время жизни (TTL) от 30 минут до 7 дней и автоматически истекают. Доступ к принтеру осуществляется только для чтения через Moonraker API — никаких команд управления не передаётся.

## Цели

1. Предоставить оператору инструмент для безопасного расшаривания статуса 3D-печати заказчикам
2. Исключить возможность управления принтером со стороны заказчиков
3. Обеспечить автоматическое истечение временных ссылок
4. Предоставить интуитивно понятный интерфейс просмотра без необходимости регистрации или установки ПО
5. Работать в локальной сети с возможностью внешнего доступа через Caddy reverse proxy

## User stories

### Оператор 3D-принтера
- Как оператор, я хочу генерировать временную ссылку на просмотр статуса печати, чтобы отправить её заказчику
- Как оператор, я хочу настраивать время жизни ссылки (30 мин — 7 дней), чтобы ограничить доступ
- Как оператор, я хочу указывать опциональный комментарий при генерации ссылки (например, "Заказ Иванова - бенчерг")
- Как оператор, я хочу видеть список всех активных ссылок с их сроком действия и комментариями
- Как оператор, я хочу отзывать ссылки досрочно, если нужно прекратить доступ
- Как оператор, я хочу копировать ссылку и видеть QR-код для быстрой отправки

### Заказчик
- Как заказчик, я хочу открыть ссылку в браузере без регистрации и установки ПО
- Как заказчик, я хочу видеть актуальный прогресс печати в процентах и слоях
- Как заказчик, я хочу видеть визуализацию G-code с текущим положением печатающей головки
- Как заказчик, я хочу видеть температуры сопла и стола в реальном времени
- Как заказчик, я хочу знать оставшееся время печати

### Оператор (дополнения)
- Как оператор, я хочу указывать опциональный комментарий при генерации ссылки (например, "Заказ Иванова - бенчерг")
- Комментарий отображается в списке активных ссылок для удобной идентификации

## Основные сценарии

### 1. Генерация временной ссылки оператором

1. Оператор открывает админ-панель (доступна только из локальной сети)
2. Выбирает время жизни ссылки (TTL) из предустановленных вариантов (30 мин, 1 час, 1.5 часа, ..., 24 часа) или вводит произвольное значение (шаг 30 мин)
3. Нажимает кнопку "Сгенерировать ссылку"
4. Система генерирует уникальный токен (32 символа [a-zA-Z0-9])
5. Система сохраняет токен в базу данных с привязкой к времени истечения
6. Оператору отображается ссылка вида `https://print.example.com/view/{token}` и QR-код
7. Оператор копирует ссылку или сканирует QR-код и отправляет заказчику

### 2. Просмотр статуса печати заказчиком

1. Заказчик переходит по ссылке в браузере
2. Система валидирует токен (проверяет наличие и срок действия)
3. Если токен валиден — загружается страница просмотра
4. Страница подключается к WebSocket для real-time обновлений
5. Отображается:
   - Название файла печати
   - Прогресс-бар с процентами
   - Текущий слой / общее количество слоёв
   - Время печати и ETA
   - Статус принтера (printing, paused, complete, error, standby)
   - Температуры сопла и стола (текущая / целевая)
   - Визуализация G-code с подсветкой текущего слоя
6. Данные обновляются автоматически при изменении состояния принтера

### 3. Истечение временной ссылки

1. При каждом запросе к странице или API система проверяет `expiresAt` токена
2. Если текущее время превышает `expiresAt` — доступ блокируется
3. Пользователю отображается сообщение "Link expired"
4. Оператор может сгенерировать новую ссылку с любым TTL

### 4. Отзыв ссылки оператором

1. Оператор открывает админ-панель → список активных ссылок
2. Нажимает кнопку "Отозвать" рядом с нужной ссылкой
3. Система помечает токен как отозванный в базе данных
4. При следующем запросе по этой ссылке доступ блокируется

## Успех / метрики

### Функциональные критерии
- Страница просмотра отображает корректные данные о прогрессе, температурах, времени печати
- Визуализатор G-code отрисовывает траекторию печати, подсвечивает текущий слой, позволяет переключаться между слоями
- Временные ссылки истекают автоматически по истечении TTL
- WebSocket обеспечивает real-time обновление данных без перезагрузки страницы
- Админ-панель позволяет генерировать ссылки с любым TTL (шаг 30 мин)
- QR-код корректно кодирует ссылку и сканируется мобильными устройствами

### Метрики качества
- Время загрузки страницы просмотра < 2 секунд
- Задержка обновления данных через WebSocket < 1 секунды
- Ссылка недоступна после истечения TTL (410 Gone)

### Критерии приёмки
- Docker-контейнер запускается и работает стабильно
- Caddy корректно маршрутизирует запросы (публичный доступ к `/view/*`, приватный к `/admin/*`)
- При потере связи с Moonraker отображается индикатор переподключения
- Логирование запросов к админ-панели работает для аудита

## Ограничения и допущения

### Технические ограничения
- Сервис работает в Docker-контейнере на VPS в локальной сети
- Требуется доступ к Moonraker API по HTTP (`http://192.168.1.XXX:7125`)
- Для внешнего доступа требуется Caddy reverse proxy с HTTPS
- Moonraker должен быть доступен из сети VPS
- База данных токенов — SQLite (better-sqlite3)

### Ограничения доступа
- Админ-панель доступна только из локальной сети (защита через Caddy по IP)
- Авторизация в админ-панели не реализуется (предполагается сетевая изоляция)
- Просмотр страницы требует только валидного токена, без регистрации

### Инфраструктура
- Caddy reverse proxy установлен на отдельном сервере (не в Docker Compose этого проекта)
- Требуется отдельная конфигурация Caddyfile для маршрутизации запросов

### Ограничения TTL
- Минимальное значение: 30 минут
- Максимальное стандартное значение: 24 часа
- Произвольное значение: до 7 дней (10080 минут)
- Шаг кратности: 30 минут

### Допущения
- В сети не более одного активного принтера (или нужно расширить для нескольких)
- Moonraker API стабилен и доступен во время работы сервиса
- QR-код сканируется стандартными камерами мобильных устройств

## Риски

### Технические риски
1. **Потеря связи с Moonraker** — при недоступности API принтера данные не обновляются, страница показывает устаревшую информацию
   - Митигация: отображать индикатор переподключения, хранить последнее известное состояние

2. **Истечение всех активных ссылок** — если оператор забыл продлить доступ, заказчик теряет возможность просмотра
   - Митигация: оператор может генерировать новые ссылки в любой момент, QR-код позволяет быстро получить доступ

3. **Коллизия токенов** — теоретическая вероятность генерации одного токена (пространство 62^32)
   - Митигация: использование криптографически стойкого генератора, уникальный индекс в БД

### Безопасность
1. **Перехват ссылки** — если ссылка передана по незащищённому каналу, третьё лицо может получить доступ
   - Митигация: HTTPS через Caddy, короткое время жизни ссылок

2. **Сканирование токенов** — возможность перебора токенов для получения доступа
   - Митигация: пространство 62^32 делает перебор нереалистичным, rate limiting на API

3. **Отказ в обслуживании** — массовые запросы могут перегрузить сервер или Moonraker
   - Митигация: rate limiting, кэширование статуса принтера

### Эксплуатационные риски
1. **Потеря базы данных токенов** — при краше контейнера без персистентности все ссылки станут недействительны
   - Митигация: volume mount для `/data/tokens.db`

2. **Ошибки в конфигурации Caddy** — неправильная маршрутизация может открыть админ-панелю публичный доступ
   - Митигация: документация, проверка конфигурации перед развёртыванием

## Открытые вопросы

1. **Поддержка нескольких принтеров** — требуется ли возможность выбирать конкретный принтер при генерации ссылки? Сейчас TZ описывает один принтер.

2. **Привязка токена к файлу печати** — нужно ли ограничивать видимость токена только текущим файлом или ссылка показывает всегда актуальный статус принтера?

3. **Хранение истории ссылок** — нужно ли хранить историю отозванных и истёкших ссылок для аудита, или удалять их из БД?

4. **Уведомления об истечении** — нужно ли уведомлять оператора о том, что ссылка скоро истечёт (например, за 30 минут)?

5. **Автоматическая генерация ссылок** — нужно ли настраивать автоматическую генерацию ссылки при старте печати с заданным TTL?

6. **Формат QR-кода** — нужен ли QR-код с TTL-счётчиком (показывает сколько осталось времени жизни)?

7. **Логирование** — какие именно события нужно логировать (доступ к админ-панели, генерация ссылок, просмотры, отозванные ссылки)?
